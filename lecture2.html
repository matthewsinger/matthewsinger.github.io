The Docs

The docs, which is short for documentation, tells us about our language and what functions are pre-defined. All of the programs we write will be about composing, which is a fancy word for combining, the pre-defined functions in ways that suit our purposes for a specific program.

To access the docs, we can go Help > Racket Documentation which will work even if we don’t have internet access. With internet access, we can just google "BSL docs". Let’s look at an example we’re familiar with: substring. Looking at the docs, it says it takes three inputs, a string and two natural numbers, and outputs a string.

What is a natural number? Why can’t this just say number? (a whole number >=0, a decimal or negative doesn’t make sense in this context)

It not only tells us the type of inputs, but how the function behaves, with an example.

Warning: make sure you are in the right docs. If I type + into the search box at the top of the page, I get 14 different results. DrRacket supports many languages, and you have to make sure you’re looking at the correct one. Make sure the + you click is next to lang/htdp-beginner. Then, click +.

If you ever have a question about BSL, the docs are a good place to look.

Reusing images

So, going back to one of the examples of combining images we did, what if we wanted to do the reverse?

For example, if we need to flip the order of the arguments passed to above, we need to keep writing the same thing over again. This is tedious at best and error prone at worst.

So how do we reuse something like (rectangle 100 50 "solid" "orange") without having to type it out over and over? We define it as a constant, like this:

(define ORANGE-RECTANGLE (rectangle 100 50 "solid" "orange"))

Now, when we press run, we can use this constant in the interactions window and see it returned, or we can use it to build up other constants.

(define ORANGE-OVER-GREEN (above (rectangle 100 50 "solid" "orange") ORANGE-RECTANGLE)).

We use CAPS for constants for stylistic reasons.

Let’s use constants to build up a simple, clean image: a sunset. (draw basic shape on board so students know the goal)

(define WIDTH 400)
(define HEIGHT 300)
(define BG (rectangle WIDTH HEIGHT "solid" "blue"))
(define GRASS (rectangle WIDTH (/ HEIGHT 2) "solid" "green"))
(define SUN (circle (/ WIDTH 4) "solid" "yellow"))

We have our components. How do we combine them? Let’s look in the docs. (based on student suggestions/verbiage, use overlay/align or place-image)

(define SUNSET (overlay/align "middle" "bottom" GRASS (overlay SUN BG)))

Notice how our computation became very clean when we used constants. If we wanted to change the width of the image, for example, all we’d have to do is change WIDTH. (demonstrate)

Code Quality

This is going to be our first discussion of what makes for good code. This is going to be a massive focus for us this year. Let’s demonstrate why.

Look what happens we expand this out (what the image would look like without constants):

(overlay/align "middle" "bottom" (rectangle 400 (/ 400 2) "solid" "green")
               (overlay (circle (/ 400 4) "solid" "yellow")
                        (rectangle 400 300 "solid" "blue")))

Look at how much worse that is. Changing the width would require changes in now 4 places instead of 1, and it would be very easy to make a mistake. It’s also so hard to tell what this would produce. Constants make code more less error-prone and more readable. The most powerful tool you have as a programmer is English: use it!

Indentation is also crucial. We can make this code even worse:

(overlay/align "middle"        "bottom"
               (rectangle 400 (/ 400 2)
                          "solid" "green")
               (overlay (circle (/ 400 4)
                                "solid" "yellow")
                        (rectangle 400 300 "solid" "blue")))

Now it’s even less clear what is being passed to which function.

TIP: Use Racket > Reindent All to properly indent your code. This is a lifesaver, and should ALWAYS be done before submitting code.
