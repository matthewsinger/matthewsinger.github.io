The abstraction game

Ask for two student volunteers to come to the whiteboard. Ask one student to define a function which determines if dog is in a list of strings, and the other to define a function which determines if cat is in a list of strings.

Now those two look awfully similar. In fact, we’ve written an awful lot of similar looking code so far, and it feels very repetitive. So now we’re going to talk about how to abstract that code. Let’s look at the body of this code and see what’s actually different. Just the strings themselves (and the names of the recursive calls, but they’re in the same place, so it’s basically the same). Let’s highlight those.

Whatever we highlight we make a parameter to our new code.

So we go from

; cat-in? : LoS -> Boolean
; Is cat in los?
(define (cat-in? los)
  (cond [(empty? los) #f]
        [(cons? los) (or (string=? (first los) "cat") (cat-in? (rest los)))]))
(check-expect (cat-in? empty) #f)
(check-expect (cat-in? (list "dog")) #f)
(check-expect (cat-in? (list "dog" "cat")) #t)
 
; dog-in? : LoS -> Boolean
; Is dog in los?
(define (dog-in? los)
  (cond [(empty? los) #f]
        [(cons? los) (or (string=? (first los) "dog") (dog-in? (rest los)))]))
(check-expect (dog-in? empty) #f)
(check-expect (dog-in? (list "cat")) #f)
(check-expect (dog-in? (list "cat" "dog")) #t)

To

; in? : LoS String -> boolean
; Is s in los?
(define (in? los s)
  (cond [(empty? los) #f]
        [(cons? los) (or (string=? s (first los))
                         (in? (rest los) s))]))
(check-expect (in? empty "foo") #f)
(check-expect (in? (list "bar") "foo") #f)
(check-expect (in? (list "bar" "foo") "foo") #t)

And even better, we can define our original functions in terms of them. This leads to less mistakes when we are sure our abstractions are correct.
; cat-in? : LoS -> Boolean
; Is cat in los?
(define (cat-in? los)
  (in? los "cat"))
(check-expect (cat-in? empty) #f)
(check-expect (cat-in? (list "dog")) #f)
(check-expect (cat-in? (list "dog" "cat")) #t)
 
; dog-in? : LoS -> Boolean
; Is dog in los?
(define (dog-in? los)
  (in? los "dog"))
(check-expect (dog-in? empty) #f)
(check-expect (dog-in? (list "cat")) #f)
(check-expect (dog-in? (list "cat" "dog")) #t)

And it becomes even easier to define new functions!
; bee-in? : LoS -> Boolean
; Is bee in los?
(define (bee-in? los)
  (in? los "bee"))
(check-expect (cat-in? empty) #f)
(check-expect (cat-in? (list "bee")) #f)
(check-expect (cat-in? (list "cat" "bee")) #t)

But what about sum and product? Let’s define them.
; sum : LoN -> Number
; Sum a list of numbers
(define (sum lon)
  (cond [(empty? lon) 0]
        [(cons? lon) (+ (first lon) (sum (rest lon)))]))
(check-expect (sum empty) 0)
(check-expect (sum (list 1 2 3 4)) 10)
 
; product : LoN -> Number
; Multiply a list of numbers
(define (product lon)
  (cond [(empty? lon) 1]
        [(cons? lon) (* (first lon) (sum (rest lon)))]))
(check-expect (sum empty) 1)
(check-expect (sum (list 1 2 3 4)) 24)

What are the differences here? The base case... and function. But how could we get around that; we can’t pass other functions to functions. Can we?

ISL

Everyone, stand up again. You have just graduated to ISL.

So how do we abstract this function?

; collapse : LoN ? Number -> Number
; Collapse lon into one value with f onto b
(define (collapse lon f b)
  (cond [(empty? lon) b]
        [(cons? lon) (f (first lon) (collapse (rest lon) f b))]))
(check-expect (collapose (list 1 2 3) max 0) 3)
 
(define (sum lon) (collapose lon + 0))
(define (product lon) (collapose lon * 1))

But what type do we give f? We annotated function types with [Input... -> Output], so:

; collapse : LoN [Number Number -> Number] Number -> Number
; Collapse lon into one value with f onto b
(define (collapse lon f b)
  (cond [(empty? lon) b]
        [(cons? lon) (f (first lon) (collapse (rest lon) f b))]))
(check-expect (collapose (list 1 2 3) max 0) 3)
 
(define (sum lon) (collapose lon + 0))
(define (product lon) (collapose lon * 1))

All of the tests should still pass.
