Lecture 6: Conditional Programming

Motivation: We want to be able to change the behavior of our code based on input

Topics to cover: boolean, if, cond, what is and isn’t evaluated

Recap

Any insights from the homework? What didn’t people understand?

Big number

Let’s get a cutoff for a big number. (notes assume it’s 50)

Great. We want a function that, given a big number, will output "big", and will otherwise output "small". Let’s start designing that function. Let’s make sure to write tests below, at, and above the cutoff.
; describe : Number -> String
; Describe x as big or small
(define (describe x)
  "")
(check-expect (describe 49) "small")
(check-expect (describe 50) "big")
(check-expect (describe 51) "big")

So what do we do now? What we want to do is "if x is bigger than or equal to 50, output big, otherwise, output small."

But right now, we have no way to do that. Let’s change that.

Comparing numbers

Let’s start by inspecting the "bigger than or equal to 50." How would you express that in algebra? As x >= 50. Great. Here’s how to express that in code (say x is 80).

If we run (>= 80 50) in the interactions window, it outputs #true. This doesn’t look like anything we’ve seen before, and we now the interactions window only produces values. So #true must be a value. Enter booleans.

Booleans

Booleans are a new type of value, and there are only two of them: #true and #false. We can also write #t or #f. Let’s experiment with some more functions that output booleans.

(= 1 2)

(> 1 2)

(>= 1 2)

(<= 1 2)

(< 1 2)

But these are only number comparisons. Are there any others?

(string=? "cat" "dog")

(string=? "cat" "cat")

(string? "cat")

(string? 1)

(number? 1)

(number? "cat")

(boolean? #t)

There are many other functions that output booleans, and we’re going to write many of our own throughout the year.

How many values do we have now? What are they? (4: number, string, image, boolean)

if

Let’s go back to our describe function, and what it is we want to do. "if x is bigger than or equal to 50, output big, otherwise, output small."

We now know how to do part of that, but the first word is still new: if.

As it turns out, if is a keyword in BSL, much like define, and it has special properties. An if expression has very similar syntax to a function call, and has 3 components. You can almost think of if as a function; we’ll explain why it isn’t one shortly.

If takes a test expression, a then expression, and an else expression. The test expression must evaluate to a boolean.

Here are two simple examples (interactions window):

(if #t 4 "cat")

(if #f 4 "cat")

As you can see, it evaluates the then expression if the test is true, and the else expression if it’s false.

What do you think will happen if we run (if #t 4 (/ 5 0))? Let’s try it. It doesn’t break! if won’t even evaluate the expression it isn’t going to produce. This is why if is special, because it doesn’t follow the in-to-out rules of evaluation like function calls do. Let’s confirm this in the stepper, and by the halloween text when we hit run in the definitions window.

So, we now know if. Let’s finish describe.

; describe : Number -> String
; Describe x as big or small
(define (describe x)
  (if (>= x 50) "big" "small"))
(check-expect (describe 49) "small")
(check-expect (describe 50) "big")
(check-expect (describe 51) "big")

Let’s tweak the code so we have three categories: huge or big or small.

; describe : Number -> String
; Describe x as big or small
(define (describe x)
  (if (>= x 500) "huge" (if (>= x 50) "big" "small")))
(check-expect (describe 49) "small")
(check-expect (describe 50) "big")
(check-expect (describe 51) "big")
(check-expect (describe 499) "big")
(check-expect (describe 500) "huge")
(check-expect (describe 501) "huge")

Well this is getting harder to read very quickly. Can you imagine what four categories would look like? if is fine if we only want our code to behave in one of two ways, but after three or more conditional choices, it starts to get ugly, fast. Enter cond.

cond

cond takes if and makes it much more robust. Here is what a cond looks like, compared to if.

(if test then else)
(cond [test1 answer1]
      [test2 answer2]
      [test3 answer3])

And so on and so forth. Let’s tweak "describe" to use it.
; describe : Number -> String
; Describe x as big or small
(define (describe x)
  (cond [(>= x 500) "huge"]
        [(>= x 50) "big"]
        [#true "small"]))
(check-expect (describe 49) "small")
(check-expect (describe 50) "big")
(check-expect (describe 51) "big")
(check-expect (describe 499) "big")
(check-expect (describe 500) "huge")
(check-expect (describe 501) "huge")

We always want to return "small" if x doesn’t pass either of the above conditions, so it makes sense to use #true. However, this looks weird. cond comes with an optional special keyword for the last clause: else. Let’s use that instead.

; describe : Number -> String
; Describe x as big or small
(define (describe x)
  (cond [(>= x 500) "huge"]
        [(>= x 50) "big"]
        [else "small"]))
(check-expect (describe 49) "small")
(check-expect (describe 50) "big")
(check-expect (describe 51) "big")
(check-expect (describe 499) "big")
(check-expect (describe 500) "huge")
(check-expect (describe 501) "huge")

But what happens if we don’t have an else?

No backup

Let’s design a function that describes some foods as tasty or gross.

; describe-food : String -> String
; Describe food as either tasty or gross
(define (describe-food food)
  (cond [(string=? food "apple") "tasty"]
        [(string=? food "mold") "gross"]))
(check-expect (describe-food "apple") "tasty")
(check-expect (describe-food "mold") "gross")

But what happens when you run (describe-food "orange")? It blows up! Always be sure your cond can handle any input that fits with your signature and purpose statement.
