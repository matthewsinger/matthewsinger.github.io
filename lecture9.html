Lecture 9: Unions

Motivation: We want to represent complex data.

Topics to cover: Union data definitions, interpretations, examples, templates, code

Recap

Any insights from the homework? What didn’t people understand?

Union data

Let’s explore a function we used on a recent homework, string->number. What is its signature?

; string->number : String -> Number... or #f?

That’s clearly not a good data type. To properly capture the signature of string->number, we need to expand beyond primitive types and start defining our own.

; A MaybeNumber is one of:
; - #f
; - Number

This says a MaybeNumber is either #false or a Number. This kind of data type is a union data type, which means it is the union of two or more types of data. Like unions in math, which operate over sets, a union data type is the type which contains all of the elements from each of the data types it is a union over.

As seen by MaybeNumber, we can define union types by other types as well as individual values.

Now that we can define our own data definitions, much like our own functions, we need a systematic process to understand and use them.

The Design Recipe For Data

The design recipe for data has four steps: data definition, intepretation, examples, and template.

We’ve already completed the data definition for MaybeNumber. A data definition is a comment which defines a data type, and that’s completed.

An interpretation is something for the reader that clarifies what the data represents. Let’s add one.

; A MaybeNumber is one of:
; - #f
; - Number
; and represents either some non-number value or a number

Here, the interpretation lets us know a #f represents a non-number. That’s why (string->number "sheep") produces #f.

Next up is examples. We define examples so we both know our data definition is well formed and to have easy data to test on.

; A MaybeNumber is one of:
; - #f
; - Number
; and represents either some non-number value or a number
(define maybenum-f #f)
(define maybenum-10 10)

The last step is a template. A template tells us how to write code that processes this kind of data. For now, when writing a template, we just need to ask how to tell the types of MaybeNumbers apart from each other.
; A MaybeNumber is one of:
; - #f
; - Number
; and represents either some non-number value or a number
(define maybenum-f #f)
(define maybenum-10 10)
 
; maybenum-temp : MaybeNumber -> ?
(define (maybenum-temp mn)
  (cond [(false? mn) ...]
        [(number? mn) ...]))

The signature of a template is always that data type to ?, because we don’t know what kind of function it will be used for. As our union data types have more than one kind of thing it could be, so we need to differentiate the cases with a cond.

For now, when writing a template, the two questions to ask yourself is:

  1. How many cases of the data are there?

  2. How do I tell them apart?

The ...’s indicate a "fill in the blank". If your template is properly formed (i.e. is a proper function) then BSL can still run. However, it still produces halloween text, because it sees a cond that hasn’t been run.

To fix this, we can either add ...’s before the cond, like this:

; maybenum-temp : MaybeNumber -> ?
(define (maybenum-temp mn)
  (... (cond [(false? mn) ...]
             [(number? mn) ...])))

or surround it with a block comment, whichever you prefer.

It used to be the case you could put #; in front of the define to nicely comment out the whole template and still maintain syntax highlighting, but that doesn’t seem to be the case in the student languages anymore?

Now, let’s see how these steps help us design a function. Let’s write one that takes a MaybeNumber and produces a MaybeNumber by adding 10 to it, if it’s a number, and otherwise leaves it as is.

; add10 : MaybeNumber -> MaybeNumber
; Add 10 the number (if it is there)
(define (add10 mn)
  #f)
(check-expect (add10 maybenum-f) maybenum-f)
(check-expect (add1 maybenum-10) 20)

Look at how easy that was to test because of the examples we already had lying around! Now, let’s fill in the body of the function by copy/pasting the template from our data definition.

; add10 : MaybeNumber -> MaybeNumber
; Add 10 the number (if it is there)
(define (add10 mn)
  (cond [(false? mn) mn]
        [(number? mn) (+ 10 mn)]))
(check-expect (add10 maybenum-f) maybenum-f)
(check-expect (add1 maybenum-10) 20)

So easy! This is why templates are invaluable. We write them once and can reuse them however many times we want.

Let’s demonstrate this by writing another function, mn-or, which takes a MaybeNumber and a backup number, and produces the MaybeNumber if it’s there and the backup if it’s not.
; mn-or : MaybeNumber Number -> Number
; mn if it's there, else backup
(define (mn-or mn backup)
  (cond [(false? mn) backup]
        [(number? mn) mn]))
(check-expect (mn-or maybenum-f 5) 5)
(check-expect (mn-or maybenum-10 5) maybenum-10)

One other example

Given time, work through this example:

; An ImperialMeasure (IM) is one of:
; - "inch"
; - "foot"
; - "yard"
 
(define im-inch "inch")
(define im-foot "foot")
(define im-yard "yard")
 
; im-temp : IM -> ?
(define (im-temp im)
  (... (cond [(string=? im im-inch) ...]
             [(string=? im im-foot) ...]
             [(string=? im im-yard) ...])))
 
; to-inches : Number IM -> Number
; Given a measurement, convert it to inches
(define (to-inches n im)
  (cond [(string=? im im-inch) n]
        [(string=? im im-foot) (* n 12)]
        [(string=? im im-yard) (* n 36)]))
(check-expect (to-inches 20 im-inch) 20)
(check-expect (to-inches 20 im-foot) 240)
(check-expect (to-inches 20 im-yard) 720)
