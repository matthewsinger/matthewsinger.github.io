Game Part 1

Programming Language BSL+

Purpose Design a large program.

Setup

Remember to change your language to Beginning Student Language with List Abbreviations. If you’re more comfortable using cons instead of list to define your examples and tests you’re still welcome to do so.

This homework is going to have a lot of text to describe a program that is more complex than any we’ve made in class before. However, you do not have to complete the game for this homework. The game is going to be broken up over four homeworks:

  1. Designing the game, including constants, data definitions, the main function, and handler signatures/purpose statements

  2. Drawing the game

  3. User interaction and ending the game

  4. How the game advances over time

Now, for the description of the game.

Type-o-Rama!

Type-o-Rama! is a type-training video game in which words fall from the sky and stack up as they reach the bottom. If you enter a word as it falls the word is cleared from the screen. Once it has reached the bottom, though, it is stuck there. The game ends when a newly-generated falling word overlaps with a stuck word. Here is a link to a sample video of gameplay.

Specs

Below are specifications for the game:

  • The game will be launched with a main function that takes in how many seconds pass between ticks.

  • The output will be the score of the game which is the inverse of how many seconds pass between ticks times how many ticks passed. This is essentially "how long the game lasted times how impressive it was". Words falling at a slower rate is less impressive, which is why we take the inverse of it.

  • Every other tick a new word is generated at the top of the screen. A word should not generate every tick.

  • When a new word is generated, it should always be at the top of the screen. Its position on the x-axis should be random, though. Just be sure the whole word fits on the screen.

  • The text of the newly generated word will should be randomly selected from some constant list of words.

  • Words should always get stuck right before they go out the bottom of the gameplay section of the screen or before they overlap with a word already stuck.

  • Falling and moving words should be shown with a different color.

  • Every tick falling words should fall one row.

  • The player should be able to see what they are typing.

  • The player should be able to fully interact with the text they are typing, as described in homework 14.

  • Hitting "enter" should trigger falling word elimination and should also clear what the player was typing. All falling words that exactly match what the player was entering should be eliminated.



Hints:

  • What happens over time is the most complex part of this game. Break down your on-tick function so that each helper is in charge of one task. Take care in the order you call the helper functions, too; falling words shouldn’t move before checking whether or not they need to become stuck words.

    When designing a complex on-tick with, for example, 3 steps, it helps to use the following structure:
    ; update-world-over-time : <SomeWorld> -> <SomeWorld>
    ; do step 1, then step 2, then step 3
    (define (update-world-over-time world)
      (step3
       (step2
        (step1 world))))
     
    ; step1 : <SomeWorld> -> <SomeWorld>
     
    ; step2 : <SomeWorld> -> <SomeWorld>
     
    ; step3 : <SomeWorld> -> <SomeWorld>

    Of course, the functions should be better named and have purpose statements. The point is that if each helper has this signature, it’s very easy to order the steps and combine them.

  • The following two functions should appear on your wishlist:
    ; maybe-generate-new-moving-word : <YourWorldDefintion> -> <YourWorldDefintion>
    ; Maybe generate a new moving word if it's the appropriate time
     
    ; generate-new-moving-word/text : <YourWorldDefintion> String -> <YourWorldDefintion>
    ; Generate a new moving word with this text

  • explode is a function that breaks up a string up into its list of 1Strings (strings of size 1) and should be useful when rendering a word.

  • big-bang clauses work independently of each other. When writing the on-tick clause, only think about what changes over time. When writing the to-draw clause, only think about how to render the world. When writing the on-key clause, only think about how the world changes when someone enters a key. When writing the stop-when clause, only think about what causes the game to stop.

  • If all of your words exist in a grid system, where each letter occupies one space in the grid, it will be much easier to detect which words should become stuck.

For this homework

For this homework, you should define your graphical constants and complete your data definitions (along with intepretations, examples, and templates, of course). You should then complete your main function, and give all of your big-bang handlers a signature and purpose statement.

You should start your wishlist with the two functions mentioned in the hints above. You may optionally extend your wishlist with further specifics, but it is not required.
